# Database

PostgreSQL is the source of truth for MedTech ERP. The API accesses it through Kysely, using generated TypeScript types in `database/generated/db.ts`.

## Local Commands

```bash
npm run db:up
npm run db:wait
npm run db:migrate
npm run db:seed
```

To rebuild local data from scratch:

```bash
npm run db:reset
```

To regenerate Kysely database types after applying schema changes:

```bash
npm run db:types
```

## Migration Order

Migrations are SQL files and must be applied in order:

1. `database/migrations/0001_core.sql`
2. `database/migrations/0002_business.sql`
3. `database/migrations/0003_hr_leave.sql`

The npm `db:migrate` script applies this exact order with `psql` and `ON_ERROR_STOP=1`.

## Seed Order

Seeds are also ordered:

1. `database/seeds/0001_roles_permissions.sql`
2. `database/seeds/0002_master_data.sql`

The first seed creates baseline roles, permissions, role grants, and document sequences. The second seed creates company/master data and the local `admin` super admin profile.

## Schema Conventions

- IDs use PostgreSQL UUIDs generated by `gen_random_uuid()`.
- Many operational tables use `created_at`, `updated_at`, and optional `deleted_at`.
- Soft deletion is represented by `deleted_at`, not physical deletion, unless a table explicitly uses cascade behavior for dependent workflow rows.
- `public.set_updated_at()` keeps `updated_at` current through triggers.
- `public.audit_row_change()` records immutable audit rows for selected business tables.
- `public.document_sequences` and `public.next_document_number(entity)` generate prefixed business numbers.
- Enums and check constraints enforce important status/date rules at the database layer.

## Generated Types

`database/generated/db.ts` is generated by `kysely-codegen` from the live schema.

Rules:

- Do not hand-edit `database/generated/db.ts` as the source of truth.
- Add or update SQL migrations first.
- Apply migrations locally.
- Run `npm run db:types`.
- Commit the migration and regenerated type changes together when schema changes are intended.

## Permission Data

Core authorization data lives in:

- `roles`
- `permissions`
- `role_permissions`
- `profiles`
- `user_roles`

The API loads these through `ProfilesService` and enforces them with `PermissionGuard`. UI hiding is not the authorization boundary; API permissions and database constraints are the enforceable layer currently present in the app.

## HR Leave Data

The connected HR leave workflow uses:

- `employee_leave_balances`
- `leave_applications`
- `leave_approvals`
- `leave_handovers`
- `leave_clearances`
- `leave_rejoins`

`0003_hr_leave.sql` also creates triggers that:

- Validate date ranges, overlapping approved leave, and balance constraints.
- Create or update pending approval rows when applications are submitted.
- Apply approval decisions back to applications and leave balances.
- Create or update handover, clearance, and rejoin records from approved leave applications.
- Normalize rejoin status from actual return dates.

## Docker Database Notes

The full `docker-compose.yml` stack initializes a `medtech` database when its volume is first created. The local-only `docker-compose.local.yml` stack creates `medtech_local` and expects you to run `npm run db:migrate` and `npm run db:seed`.

Both expose PostgreSQL on host port `54322`; do not run both stacks at the same time unless you intentionally change ports or container names.
